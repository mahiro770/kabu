# 株式AI分析ツール 設計書

最終更新: 2026-07-18

## 1. システム概要

日本株・米国株を対象としたAI支援の株式分析Webアプリケーション。Streamlitのマルチページ機能を用いて構築されており、ローソク足チャート・多数のテクニカル指標・財務指標・AIによる自然文の分析レポート・ニュース駆動の銘柄スクリーニング・条件（時価総額/PER/PBR等）による銘柄検索・チャートパターンの学習コンテンツ・複数階層のウォッチリスト機能を1つのアプリとして提供する。

- **公開URL**: https://kabusearch.streamlit.app/
- **リポジトリ**: https://github.com/mahiro770/kabu
- **想定利用者**: 個人投資家（開発者本人およびURLを共有した知人・コミュニティ）
- **表示言語**: 日本語デフォルト、一部ページは英語表示にも対応（`app.py`のみ。`pages/`配下は日本語固定）

### 1.1 提供価値

| 課題 | 提供する解決策 |
|---|---|
| 個別銘柄のテクニカル・財務状況を素早く俯瞰したい | 1画面でローソク足・8種のシグナル・財務指標をまとめて表示 |
| 大量の指標を自分で解釈するのが大変 | ローカルLLM/クラウドLLM（Gemini・Groq）による自然文レポート生成 |
| 日々のニュースから注目銘柄を探すのが大変 | ニュース見出しをAIに要約・ランキングさせるスクリーニング機能 |
| 時価総額・PER・配当利回りなどの条件で銘柄を絞り込みたい | Yahoo!ファイナンス公式スクリーニングAPIによる条件検索（中小型株も対象） |
| チャートパターンの意味が分からない | サンプルチャート付きの解説＋自分のウォッチリスト銘柄で該当パターンを検出 |
| 気になる銘柄を仲間と共有したい | 合言葉制の個人/コミュニティ/公開の3階層ウォッチリスト |

### 2.1 実行モデル

Streamlitはファイルベースのマルチページアプリで、ユーザー操作のたびに該当ページのPythonスクリプトを**最初から最後まで再実行**する（リアクティブ実行モデル）。ページ間で状態を引き継げるのは `st.session_state` のみであり、DBやファイルには依存しない。本アプリでは以下の状態を `st.session_state` に保持する。

| キー | 内容 |
|---|---|
| `watchlist` | 公開ウォッチリストの中身（起動時にFirestoreからロード） |
| `current_ticker` | 分析対象として選択中のティッカー（ウォッチリスト一覧・チャート学習・条件スクリーニングの各ページから`app.py`に`st.switch_page`で遷移する際に設定） |
| `personal_watchlist_owner` / `personal_watchlist` | 個人リストの解錠済みユーザー名とその中身 |
| `community_watchlist_owner` / `community_watchlist` | コミュニティリストの解錠済みコミュニティ名とその中身 |
| `screen_result` / `screen_names` / `screen_currency` / `screen_unit` | 条件スクリーニングの検索結果（結果内の「分析」ボタン押下によるrerunで結果が消えないよう、一時的なボタン戻り値ではなくsession_stateに保持） |

### 2.2 デプロイ構成

- ホスティング: Streamlit Community Cloud（GitHubリポジトリと連携し、`main`ブランチへのpushで自動再デプロイ）
- シークレット管理: Streamlit CloudのSecrets機能（TOML形式）に `GEMINI_API_KEY`・`GROQ_API_KEY`・`[firestore_credentials]`（GCPサービスアカウントJSONの中身）をトップレベルキーとして設定（TOMLの仕様上、テーブル見出し`[firestore_credentials]`より後ろに書くと誤ってそのテーブルの一部として読み込まれるため、他のキーより前に置く必要がある）
- ローカル開発: `.env`ファイル + `GOOGLE_APPLICATION_CREDENTIALS`（サービスアカウントJSONファイルパス）
- 認証情報の解決順序（`src/watchlist.py`）: ① `GOOGLE_APPLICATION_CREDENTIALS`のファイルパス → ② Streamlit Secretsの`firestore_credentials`テーブル → ③ デフォルトのADC（Application Default Credentials）
- デプロイ環境特有の既知事象: Streamlit CloudのGCPホスティングIPは、株探・IR BANK・Yahoo!ファイナンス等の一部サイトからボット判定されデータセンターIPをブロックされることがある（ローカル環境では問題なく取得できる）。この対策として複数の取得元を持つフォールバックチェーンを実装している（6章参照）。ホットリロード（コード変更の自動反映）が不完全でImportError等が残る場合は、管理画面から手動で「Reboot app」する必要がある

## 3. 画面構成・機能一覧

### 3.1 `app.py`（メインページ：銘柄分析）

**サイドバー**
- ウォッチリスト（銘柄名・現在値・前日比・日付軸/価格帯付きミニチャートをカード表示、クリックでその銘柄を分析）
  - ユーザー名入力
  - タブ: 🌐公開 / 🏘️コミュニティ / 👤個人（コミュニティ・個人は合言葉制、忘れた場合は名前指定で誰でも再設定可能、認証済み状態では🚪ログアウトボタンで名前・合言葉・所有権情報をクリアできる）
- 設定: AIモデル選択（🦙Ollama検出時はOllamaモデル一覧＋✨Gemini/⚡Groqが利用可能なら追加、複数プロバイダが同時に候補になり得る）、表示言語（日本語/English）、分析期間（3ヶ月〜5年）

**メインエリア**
1. 銘柄コード/会社名の入力欄＋「分析」ボタン（数字4桁は日本株`.T`として、それ以外は米国株として解決。ティッカー直接ヒットしなければ会社名検索にフォールバック）
2. ヘッダー指標: 現在値（日本株は夜間PTS株価・東証終値比も取得できた場合は併記）・52週高値/安値・RSI(14)・ボラティリティ・総合シグナルバッジ
3. タブ構成
   - **📈価格チャート**: ローソク足＋オーバーレイ選択（MA20/50/200・ボリンジャーバンド・一目均衡表・VWAP）、8指標のシグナルを角丸カードで個別表示、日経平均/S&P500との比較チャート＋期間別超過リターン
   - **📊インジケーター**: RSI・MACD・ADX・OBVの各チャート、最新テクニカル値の一覧表
   - **💹財務指標**: バリュエーション（PER/PBR/EPS/時価総額）、収益性（ROE/ROA/各種利益率）、成長性・財務健全性、信用取引（日本株のみ、複数ソースのフォールバック取得）、過去の業績推移、アナリスト業績予想・目標株価、会社概要、役員一覧、大株主一覧（同上）、関連リンク。取得元が変わった場合は「〇〇調べ」のキャプションで出典を明示し、全ソース失敗時は株探への直接リンクを案内する
   - **🤖AI分析**: ボタン押下でAIモデルがテクニカル・財務・シグナルを総合したレポートをストリーミング生成
4. 未分析時: ウォッチリスト一覧・チャート学習・使い方の各ページへのリンク＋使い方ガイド

### 3.2 `pages/1_注目銘柄スクリーニング.py`

- サイドバー: AIモデル複数選択（比較用）、応答タイムアウト（30〜300秒）、日経/Yahoo!の見出し取得件数スライダー
- 実行ボタンで日経マーケット速報＋Yahoo!ニュース経済RSSの見出しを取得し、選択した各AIモデルに並行して「翌営業日以降に注目すべき銘柄」をランキング形式で生成させる（モデルごとにタブ分け、応答が遅い/止まったモデルはタイムアウトで打ち切り、他のモデルの結果表示をブロックしない）

### 3.3 `pages/2_ウォッチリスト一覧.py`

- 共通の期間切り替え（1日/1週/1カ月/6カ月/1年/2年、1日・1週はイントラデイ足）
- 🌐公開 / 🏘️コミュニティ / 👤個人 の3タブ（コミュニティ・個人はホーム画面で解錠済みの場合のみ表示）
- 各銘柄をカード表示: 銘柄名（クリックで`app.py`に遷移し分析実行）、現在値・値動き、価格チャート（軸・日付ラベル付き）、メモ欄（数行）、フォルダ（自由入力のグループ名、保存すると同名でグルーピング）
- フォルダは選択式（ラジオボタンでフォルダを切り替え、フォルダ未設定の銘柄は「その他」に集約）

### 3.4 `pages/3_チャート学習.py`

- テクニカル指標パターン8種（MAクロス・RSI・MACD・ボリンジャーバンド・一目均衡表・ADX・OBV・VWAP）とローソク足パターン7種（陽/陰の丸坊主・十字線・ハンマー・首吊り線・陽/陰の包み足）を、それぞれサンプルチャート（実データではなく見た目重視の手作り合成データ）と解説文（買い/売りサインとしての意味）付きで一覧表示
- 「現在この形が出ている銘柄」欄: 公開・個人・コミュニティの全ウォッチリスト銘柄をスキャンし（`get_signals`/`detect_candlestick_patterns`を利用）、各パターンに現在該当する銘柄をボタンで表示。クリックで`app.py`に遷移し分析実行

### 3.5 `pages/4_使い方.py`

- 静的な使い方ガイド。アプリ概要、ホーム画面の使い方（タブ構成表）、ウォッチリストの3階層の違い、注目銘柄スクリーニング・チャート学習の概要、AIモデル（Ollama/Gemini/Groq）の違いと無料枠の注意、免責事項。各セクションから該当ページへの`st.page_link`を設置

### 3.6 `pages/5_条件スクリーニング.py`

- 市場（日本株/米国株）切り替え＋時価総額・売上高・PER・ROE・PBR・配当利回り（各下限/上限、角丸枠で項目ごとに区切ったカードUI、1刻みのステッパー）・セクターの条件を指定して検索
- yfinance公式のスクリーニングAPI（`yf.screen`/`EquityQuery`、Yahoo!ファイナンスの正規データ）を利用するため、スクレイピングではなくボット判定ブロックの心配がない。中小型株を含む数千銘柄規模から絞り込み可能（上位50件を時価総額順に表示）
- 日本株の検索結果は会社名が英語表記で返るため、`src/translate.py`の並列翻訳で日本語化（50件をThreadPoolExecutorで並列翻訳し、直列実行（約1分）を数秒に短縮）
- 各結果カードから「分析」ボタンで`app.py`に遷移

### 4.1 信用取引・大株主データの取得フォールバック

株探（kabutan.jp）はStreamlit CloudのGCP系IPをボット判定してブロックすることがある（HTTP 405 "Human Verification"）。これを緩和するため、以下のフォールバックチェーンを実装している。いずれも失敗時は例外を投げず空値を返し、呼び出し側（`app.py`）は関連リンクへの誘導メッセージで代替する。

**信用取引残高（`margin_fetcher.get_margin_trading_history`）**

1. 株探（`kabutan.jp/stock/?code=`） — 直近5週分の履歴が取れる
2. IR BANK（`irbank.net/{code}/margin`） — 同上、こちらもボット判定でブロックされることがある（CloudFront経由の403）
3. 東証公式PDF（「銘柄別信用取引週末残高」） — 全銘柄が1つの複数ページPDFにまとまっているため最新週のみ取得（`pypdf`でテキスト抽出し、銘柄コード行を正規表現でパース）。取引所公式の静的ファイル配信のためボット判定ブロックのリスクが低い

**大株主一覧（`holder_fetcher.get_major_shareholders_jp`）**

1. 株探（`kabutan.jp/stock/holder?code=`） — 有価証券報告書ベースの上位10位リスト
2. IR BANK（大量保有報告） — 5%ルール開示の変動履歴を保有者ごとに集計して直近の保有状況を推定（株探の登記ベースの上位10位リストとはデータの性質が異なる旨をUI上に注記）

いずれのソースを使ったかは取得結果に`source`キーとして保持し、UI側のキャプション（「〇〇調べ」）に反映する。

## 5. データ設計（Firestore）

Firestoreネイティブモード（Standardエディション、`asia-northeast1`）を使用。

| コレクション | ドキュメントID | スキーマ |
|---|---|---|
| `public_watchlist` | `default`固定 | `{"items": [...]}` |
| `personal_watchlists` | ユーザー名 | `{"passphrase_hash": <sha256 hex \| null>, "items": [...]}` |
| `community_watchlists` | コミュニティ名 | 同上 |

**`items`配列の各要素（銘柄エントリ）:**

```json
{
  "ticker": "7203.T",
  "name": "トヨタ自動車株式会社",
  "memo": "決算待ち",       // 任意
  "group": "自動車産業",     // 任意（未設定時はUI上「その他」に分類）
  "added_by": "山田"        // 公開・コミュニティリストのみ
}
```

- 合言葉は平文保存せず SHA-256 ハッシュ化して保存
- 未登録の名前に対する`_verify_passphrase`は常に`True`を返す（＝「初回登録＝好きな合言葉で作成可」という設計）
- 合言葉紛失時のリセットは本人確認を行わず、名前さえ分かれば誰でも再設定可能（低リスクな用途を前提とした割り切り。既存の`items`は保持される）
- ログアウトはクライアント側（`st.session_state`）のみの操作で、Firestore側のレコードには影響しない

## 7. 非機能要件・設計方針

- **エラーハンドリング方針**: 外部I/O（yfinance・スクレイピング・翻訳・AI）は例外を握りつぶし`None`/`[]`/元の値へフォールバックする。呼び出し側は常に空値を考慮した表示分岐を行う。ボット判定ブロックが疑われる外部サイトは複数ソースのフォールバックチェーンを持たせる（4.1節）
- **キャッシュ戦略**: `st.cache_data`を用途別TTLで使い分け（会社名翻訳24時間、事業概要翻訳1時間、ウォッチリストの株価取得5分、夜間PTS取得1分）。過度なAPI呼び出し・スクレイピング頻度を抑制する
- **並列処理**: 条件スクリーニング結果の会社名翻訳など、件数が多く1件ずつでは直列化して遅くなる処理はThreadPoolExecutorで並列化する
- **セキュリティ**:
  - Firestoreサービスアカウントキー・APIキーは`.gitignore`済みでリポジトリに含めない（Streamlit Secretsで管理）
  - 合言葉はハッシュ化して保存（平文非保持）
  - ただしパスフレーズ制は「なりすまし防止」レベルの軽量な仕組みであり、強固な認証基盤ではない（想定利用規模に対する意図的なトレードオフ）
  - ログアウト機能はクライアント側の`st.session_state`クリアのみで、ブラウザを閉じた場合も同様にセッションが破棄される（永続的なCookie等は使用していない）
- **国際化**: `app.py`のみ日本語/英語のインラインテキスト切り替え（i18nフレームワーク不使用）。`pages/`配下は日本語固定
- **パフォーマンス**: AIストリーミング応答は各チャンクを逐次UIに反映。ニューススクリーニングは複数モデルをタイムアウト付きスレッドで実行し、遅いモデルが全体をブロックしないようにしている。条件スクリーニングは公式APIの一括検索により、個別銘柄への逐次アクセスを避けている
- **デザイン**: オリーブグリーン（`#8aad94`）を基調としたアクセントカラーと、見出しにShippori Mincho（和文明朝体）を使用した落ち着いたトーン。Streamlitが再実行のたびに自前のテーマ色付きスタイルを後から挿入してくる箇所（見出し・プライマリボタン等）は、カスケードの後勝ちで上書きされるため`!important`で確実に上書きしている

## 8. 制約・既知の課題

- Firestoreへの書き込み失敗時のリトライ機構がない
- 合言葉紛失時の再設定に本人確認要素がなく、意図的に「誰でも再設定可能」としている（コミュニティ利用の性質上の割り切り）
- 一目均衡表の雲描画のみ、指標データフレーム自体を未来方向に拡張する例外的な実装になっている（チャート描画専用、シグナル計算には影響しない）
- Streamlitの内部DOM実装（テストID等）はバージョン間で変化するため、CSSカスタマイズの一部は複数バージョン互換のセレクタを併記している
- Ollama/Gemini/Groqすべてが利用不可の場合、AIモデル選択肢が空になりAI関連機能が使えない
- 株探・IR BANK・Yahoo!ファイナンス（夜間PTS）はいずれもスクレイピングであり、Streamlit CloudのIPがボット判定でブロックされることが断続的に発生する（ローカル環境では問題なく動作する）。フォールバックチェーンで緩和しているが、全ソースが同時に失敗する可能性は残る
- チャート学習ページの「現在この形が出ている銘柄」は、ウォッチリスト登録銘柄のみが対象（未登録銘柄からの新規発掘はできない）
- 条件スクリーニングの結果件数はAPI仕様上最大250件（現状の実装では上位50件を表示）で、全件を横断した完全な絞り込みではない
